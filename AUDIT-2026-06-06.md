# What Framework — Production-Readiness Audit & Product Review

**Date:** 2026-06-06
**Subject:** what-fw v0.9.0 (npm: `what-framework` @ 0.8.4)
**Scope:** Full smoke-audit (8 work-streams) + critical product review
**Method:** 7 parallel public/source audit agents + live browser verification. Every claim below was reproduced with a command, file:line, or URL.

---

## TL;DR

What Framework has a **genuinely excellent engine wrapped in a broken release and dishonest marketing layer.** The reactivity core, the *generated* compiler output, the router, islands, SSR primitives, and the MCP devtools are real, well-engineered, and in several places ahead of Solid/Svelte on DX. The 860-test unit suite + 40 stress tests pass (after manual cleanup), dev mode works, and a real TodoMVC ships at **8.5 KB gzip**.

But **you cannot ship a working site with it today**:

1. **A production `vite build` of any compiled-JSX app renders a blank white screen** (`useSignal() can only be called inside a component function`) — the minified core breaks component-context detection. Dev works; prod is blank. This hits the published `create-what` starter too.
2. **`npm install` fails on a clean checkout** (ERESOLVE — peerDeps pin `what-core@^0.8.4` while packages are at 0.9.0).
3. **`create-what` scaffolds `@^0.9.0` deps that aren't published** (npm latest is 0.8.4) → fresh scaffold install dies.
4. **`npm test` / `release:verify` are red on a real fresh checkout** (a stale nested `what-core@0.8.4` shadows the local `createStore` fix).

Plus a **memory leak** (component disposal never runs on list-item removal), a **CRITICAL security hole in the headline MCP feature** (devtools bridge token leaks to any website you visit), an **O(n²) compiler** on large components, and **materially false marketing claims** ("12KB runtime" is really 31KB gzip; "faster than VanillaJS" is contradicted by the repo's own 2.42× measurement).

**None of this is fatal to the project — it's a mature codebase with shallow, fixable wounds.** But it is not production-ready right now, and the public claims need to be corrected before a launch.

## Did it meet your specific goals?

> *"Ship this, fully production ready, matching/exceeding React/Svelte/Vue."*
**Not yet.** The client-side SPA story is competitive-to-better than Solid. The full-stack/SSR story (the thing that makes Next/Nuxt/SvelteKit "operational") is missing: no server data loading, no deploy adapters, server actions defined but never served. And the release path is broken (blank prod build).

> *"Build fully operational sites, better than React or Solid."*
**For client-rendered SPAs and static/island sites: yes, once the build regression is fixed.** For SSR/data-driven/full-stack sites: **no** — see the feature-parity gaps (CRIT-5 through CRIT-8).

> *"Speeds closer to or faster than VanillaJS."*
**The architecture genuinely supports it** (verified: 0 DOM elements created on a 1000-row keyed swap; real no-VDOM). **But the claim as written is false** — the only paint-inclusive measurement is 2.42× vanilla, and the "~1.03×" browser benchmark is non-credible. No framework beats hand-written vanilla; honest claim is "Solid/Preact tier."

> *"Compiler thoroughly reviewed, make sure it's fast and performant."*
**The output is fast (A). The compiler itself is not (C)** — compile time is O(n²) in elements-per-component (200→29ms, 400→78ms, 800→341ms). Large page/route files will be slow. Two specific fixes close most of it.

# Part 1 — Smoke-Audit: Issues by Severity

## CRITICAL (ship-blockers)

### C1. Production build renders a blank white screen
**Stream:** build/run · **Verified:** `npx create-what@latest x --yes && cd x && npm i && npx vite build && npx vite preview` → blank `#app`, console: `[what] useSignal() can only be called inside a component function`.
**Root cause:** the `production` export condition resolves to `dist/index.min.js`; minification breaks the component-context detection `useSignal` relies on. Forcing Vite `resolve.conditions: ['import']` (unminified) makes the identical build render correctly.
**Impact:** every deployed What app is a white screen. This is *the* showstopper.
**Fix:** make context detection minification-safe (don't rely on `Function.name`/stack identity that terser mangles). Add a CI gate that does `vite build && preview` and asserts `#app` has children — unit tests do not catch this.

### C2. Default `npm install` fails (ERESOLVE)
**Stream:** build/run · **Verified:** `npm install` at repo root → `Conflicting peer dependency: what-core@0.8.4 … peer what-core@"^0.8.4" from what-compiler@0.9.0`.
**Root cause:** packages bumped to 0.9.0 but `peerDependencies`/`dependencies` still pin `^0.8.4` (excludes 0.9.0). Only `--legacy-peer-deps` works.
**Fix:** bump all internal peer/dep ranges to `^0.9.0` (or `workspace:*`). Files: `packages/compiler/package.json`, `packages/cli/package.json`, and every package with a `what-*` peer/dep.

### C3. `create-what` scaffolds unpublishable dependencies
**Stream:** build/run, perf, public · **Verified:** `node packages/create-what/index.js app --yes && cd app && npm install` → `ETARGET No matching version found for what-compiler@^0.9.0`. npm latest for all packages is **0.8.4**; 0.9.0 is unpublished despite the `Release 0.9.0` git commit.
**Fix:** publish 0.9.0 (after C1–C2), or have `create-what` pin the latest *published* version. Add a release step that verifies a scaffold installs from the registry.

### C4. `npm test` / `release:verify` red on a clean checkout
**Stream:** build/run · **Verified:** `npm install --legacy-peer-deps && node --test examples/kanban/test/store.test.js` → `# pass 5 # fail 6`. A stale published `what-core@0.8.4` gets nested into `packages/what/node_modules/what-core` and shadows the local `createStore` action-return fix. Deleting the nested copies → `# tests 860 # pass 860 # fail 0` + stress `40/40`.
**Fix:** resolve C2 (kills the nesting), add `overrides`/`nohoist` discipline, and run `release:verify` on a truly fresh clone with no manual cleanup.

### C5. Component disposal never runs on list-item removal (memory leak)
**Stream:** core reactivity · **Verified:** a `<Row>` with `onCleanup` rendered through `mapArray({key})` then removed produced **0 cleanup calls** (expected 2). `render.js:713-721` (`_removeItemNodes`) only disposes when `n._componentCtx || n._dispose || n._propEffects` is truthy, but component ctx lives on the `c:start` comment (`_commentCtxMap`, `dom.js:328`), which has none of those flags. Unkeyed middle-removal (`render.js:515-518`) and clear-all fast paths have the same gate.
**Impact:** every list shrink/reorder leaks component effects, hook cleanups, `onCleanup`, event listeners, and devtools registrations. This is the classic bug that kills SPAs in production.
**Fix:** in `_removeItemNodes` and `reconcileList` removal paths, call `disposeTree(n)` unconditionally (or add `|| n.nodeType === 8`). Add a regression test that churns a list 1000× and asserts `mountedComponents.size` returns to baseline.

### C6. DevTools MCP bridge token leaks to any web origin → localhost takeover
**Stream:** security · **Verified:** `bridge.js:56-69` — token endpoint (`:9230 /__what_mcp_token`) responds with `Access-Control-Allow-Origin: *` and no `Origin` check; WS `verifyClient` (`:36-44`) checks only the token query param, not `Origin`.
**Attack:** while `what dev` runs, you visit any site; its JS does `fetch('http://localhost:9230/__what_mcp_token')` (CORS `*` lets it read the token), then opens `ws://localhost:9229?token=<stolen>` and drives `set-signal`/`navigate`/(with `--unsafe-eval`) arbitrary `eval` against your live app. 127.0.0.1 binding doesn't help — your browser is the confused deputy.
**Impact:** the *headline feature* (MCP devtools) is a remote-control vector. CRITICAL because dual-use and default-on in dev.
**Fix:** remove `Access-Control-Allow-Origin: *`; reject WS whose `Origin` isn't an allow-listed dev origin; prefer the file-based token handoff over an HTTP endpoint.

### C7. "12KB runtime" is false
**Stream:** perf · **Verified:** the full `what-core` index (`import from 'what-framework'`) is **92.6 KB min / 31.2 KB gzip / 27.6 KB brotli**. The "12KB" matches no real artifact (it conflates minified-not-gzipped with gzipped, assuming perfect tree-shaking of a single-feature app).
**Impact:** false claim on README line 21 + homepage + docs comparison table. Erodes trust the moment anyone measures.
**Fix:** quote measured per-import gzip numbers: full ≈31 KB gzip; tree-shaken counter ≈5.5 KB gzip; TodoMVC ≈8.5 KB gzip. (The *honest* numbers are still good — use them.)

### C8. "Faster than / near VanillaJS" is unproven and contradicted
**Stream:** perf · **Verified:** the only paint-inclusive measurement (`docs/BENCHMARK-RESULTS.md`) is **2.42× geomean vs vanilla** (select-row 10.5×). The live "~1.03×" comes from `benchmark/run-browser.js` which times *synchronous* JS with no paint, adds it to a **hardcoded** `JFB_VANILLA_REF` table, and benchmarks hand-written vanilla-style code that bypasses the public `h()` API.
**Fix:** remove/scope the claim ("compiled keyed lists are in the Solid/Preact tier"); replace the synthetic benchmark with the real js-framework-benchmark harness measuring the public API *with paint*; publish the real geomean.

## HIGH

### H1. `.map()` lowered inside a ternary/`&&` loses reactivity (compiler correctness bug)
**Stream:** compiler · **Verified:** `{o() ? t().map(...) : <li>empty</li>}` emits `_$insert(_el, o() ? _$mapArray(...) : _tmpl$2(), ...)` — **no `() =>` wrapper**, so `o()` is read once and never re-tracked. Toggling `o()` won't switch list↔fallback. `babel-plugin.js:1137-1150`. Contradicts the CHANGELOG "walks ternaries/&& correctly" claim.
**Fix:** when the lowered result isn't a bare `_$mapArray(...)` call, wrap it in `() =>`.

### H2. Compiler is O(n²) in elements-per-component
**Stream:** compiler · **Verified:** 200 els = 29 ms, 400 = 78 ms, 800 = 341 ms (warm `transformSync`); "many small components" scales linearly. Root cause: `collectSignalNamesFromScope` re-run per top-level JSXElement (`:1903`), re-enumerating all scope bindings + re-scanning function params each call (`:293-317`).
**Fix:** memoize the signal-name set once per enclosing function; lift param-prop extraction out of the binding loop.

### H3. `</script>` breakout in island state serialization (stored XSS)
**Stream:** security · **Verified:** `islands.js:87-93` returns raw `JSON.stringify(data)` embedded in a `<script data-island-stores>`; any store value containing `</script>` breaks out → stored XSS with user-controlled state.
**Fix:** escape `<`→`<` (plus `>`, `&`, U+2028/9) before embedding; export a sanctioned `serializeState()` helper.

### H4. Suspense is not wired to async (lazy → ErrorBoundary, not fallback)
**Stream:** feature-parity · **Verified:** `Suspense.onSuspend` (`components.js:88`) is never called by the renderer; `createDOM`'s catch (`dom.js:372`) routes thrown values (incl. `lazy`'s thrown promise) to `reportError`/ErrorBoundary. `createResource` never throws. So `<Suspense>`+`lazy` surfaces an error, not a fallback.
**Fix:** wire thrown promises to the nearest Suspense boundary's `onSuspend`; make `createResource` integrate with Suspense.

### H5. No server-side data loading — SSR can't render data-driven pages
**Stream:** feature-parity · **Verified:** `renderToString` (`server/src/index.js:95-142`) never awaits; no route `loader`/`load`; no data serialization into HTML for hydration. Every competitor's flagship feature is absent.
**Fix:** add async `renderToString`, a route loader concept, and a safe state-serialize/hydrate path.

### H6. Server actions defined but never served
**Stream:** feature-parity · **Verified:** `action()`/`handleActionRequest` + CSRF exist (`server/src/actions.js`) but `what dev` returns a placeholder (`cli.js:207`) and `build` emits static only. No runtime mounts `/__what_action`. RPC is non-functional out of the box.

### H7. No deployment adapters / SSR runtime
**Stream:** feature-parity · **Verified:** no Vercel/Netlify/Cloudflare/Node adapter anywhere. You can't deploy an SSR/server-action app without writing your own server.

### H8. Single throwing effect aborts the whole flush batch; ErrorBoundary doesn't catch reactive errors
**Stream:** core · **Verified:** `reactive.js:569-580` flush loop calls `_runEffect(e)` with no try/catch and `_runEffect` rethrows (`:404-407`); effects after the thrower are dropped and `_pending` is left inconsistent. The stable-effect path *does* swallow — inconsistent. Post-mount reactive errors bypass ErrorBoundary entirely.
**Fix:** wrap `_runEffect(e)` in try/catch, route to `reportError`/`__devtools.onError`, continue the batch.

### H9. Head/meta is client-only — bad SEO/SSR
**Stream:** feature-parity · **Verified:** `Head` returns `null` when `document` is undefined (`head.js:15`) and mutates `document.title` directly. SSR/static output gets no `<title>`/meta from `<Head>`. Table stakes for "production websites."

### H10. Idiomatic uncompiled `h()` + `.map()` re-creates the entire list on any change
**Stream:** perf · **Verified:** the public runtime (no compiler) re-created all 1000 elements when one item changed and on a 2-row swap. Fine-grained behavior only happens via the compiler lowering keyed `.map()` → `_$mapArray({raw:true})`. So "near vanilla" holds only for compiled JSX with keys — not the `html` tagged-template / no-build path the docs also promote.

### H11. CI badge shows failing
**Stream:** public · **Verified:** the README header badge resolves to `failing`; the 3 latest `ci.yml` runs on `main` are `cancelled`. First thing a discerning user sees.

### H12. README API ≠ scaffolded-app API
**Stream:** public · **Verified:** README uses `signal(0,'count')` + `count(c=>c+1)`; scaffolded `src/main.jsx` uses `useSignal(0)` + `count.set(c=>c+1)`. Both work, but a stranger hits two mental models on day one. (See also the **dual-API** discussion below — docs/homepage teach `useSignal`+`.set()`; README/playground/CLAUDE.md teach native `signal()`.)

### H13. ESM-only — `require('what-framework')` throws
**Stream:** public · **Verified:** `node -e "require('what-framework')"` → `ERR_PACKAGE_PATH_NOT_EXPORTED`. No `require` condition in `exports`. CommonJS tooling/SSR setups hard-fail with no documented warning.

# Part 2 — What's Working Well (don't touch)

- **Reactivity core is production-grade.** Topological-level scheduling prevents diamond glitches; iterative computed eval handles 10K-deep chains without stack overflow; hot-path reads are allocation-free and shape-stable; 10,000 writes coalesce to 2 effect runs. `reactive.js`.
- **Compiler output is best-in-class.** Static HTML hoisted to `cloneNode` templates, precise static/dynamic partitioning, per-text-node effects, document-level event delegation, clean keyed `_$mapArray` lowering with a missing-`key` warning. Verified: **0 DOM elements created on a 1000-row keyed swap** — genuine no-VDOM.
- **Router is genuinely complete for SPA** — nested layouts, dynamic + catch-all params, sync/async guards with redirect-loop detection, View Transitions, scroll restoration, prefetch-on-hover. Competitive with TanStack Router.
- **Islands architecture is a real differentiator** — 5 hydration strategies, priority queue, shared cross-island stores with SSR serialize/hydrate. More built-out than Solid's.
- **DX tooling exceeds peers** — 9-rule ESLint plugin, signal-aware error overlay, HMR with generated boundaries, full testing kit (render/fireEvent/screen/mockSignal), and a live MCP devtools bridge no competitor has.
- **Core security is solid** — default text/attr escaping, opt-in `{__html}`, `javascript:`/`data:` URL sanitization, constant-time CSRF with fail-closed defaults, correctly-sandboxed playground (`allow-scripts` only, `connect-src 'none'`), real default-off `what_eval`, clean supply chain (no postinstall scripts, no committed secrets).
- **Real-world bundles are small** — TodoMVC 8.5 KB gzip, counter 5.5 KB gzip. The honest numbers beat the invented one.
- **SSR + hydration + streaming primitives work** — `renderToString`/`renderToHydratableString`/`renderToStream` produce correct HTML and a jsdom hydration test goes SSR→interactive.
- **The new-user happy path (dev) works** — `npm create what` → install → dev → localhost:5173 with a working, compiler-transformed app, 0 vulnerabilities, ~1–4s installs.

# Part 3 — Product Review

### 1. What is it?
What Framework is a signals-based, no-virtual-DOM, "components-run-once" frontend framework (Solid-class) with a JSX→fine-grained-DOM compiler, an SSR/islands server, a router, and a distinctive **MCP devtools** server that lets AI agents inspect a running app's signals/effects/components. It is pre-launch (npm 0.8.4, repo 0.9.0), MIT, ~9K LOC core + ~2K LOC compiler, 860 unit + 40 stress tests.

### 2. Is it real?
**Yes — the software is real and substantial, but the production path is currently broken.** Internals build, the test suite is green (after cleanup), dev mode runs real apps, SSR/hydration/islands work. But a production build blank-screens (C1), `npm install` fails (C2), and the scaffolder installs nonexistent versions (C3). So: real engine, broken release. This is the line between "pre-launch" and "broken" — it's pre-launch with a regression, not vaporware.

### 3. Is it offering something unique?
**Genuinely differentiated:** (a) the **MCP devtools** — agent-queryable runtime state is a real first; no competitor has it. (b) **Islands + shared cross-island stores** more complete than Solid's. (c) **Batteries-included client surface** — router, forms+validation, SWR-style data, a11y, animation, testing, ESLint all in-tree, which Solid/Svelte push to userland.
**Not differentiated:** fine-grained signals (Solid, Svelte 5, Vue Vapor), no-VDOM (Solid, Svelte), JSX (React, Solid), SSR/islands (Astro, SolidStart, Fresh). The "AI-native" angle is the real wedge; the reactivity is table stakes.
**Verdict:** the wedge is MCP/agent-native DX, not the rendering model.

### 4. Who is this for?
**Target:** developers building AI-agent-assisted frontends who want the agent to *debug at runtime* (the MCP loop), plus solo/small teams who want a batteries-included Solid alternative. **Adjacent (if SSR ships):** content-site builders (islands) and full-stack devs (once loaders/adapters exist). **Positioning:** "Solid's performance + a batteries-included stdlib + the only agent-native devtools." Lead with the MCP angle — it's the only thing nobody else has.

**Great:** reactivity graph correctness, compiler codegen, islands, DX tooling. **Good:** router, forms, testing kit, core security. **Bad:** the build/release/packaging pipeline (F-grade — prod regression, install failure, version chaos, unpublished release). **Needs work:** memory disposal through the reconciler (C5), effect error isolation (H8), the SSR/full-stack half (H5–H7, H9), and benchmark/claim honesty (C7, C8, M7).
